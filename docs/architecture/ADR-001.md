# ADR-001 — Estratégia de Multi-tenancy

**Data:** 2026-06-02  
**Status:** Aceito  
**Autores:** Paulo (O-Tiger)

---

## Contexto

A plataforma Nexora EDU precisa servir duas instituições independentes (Instituição A — Faculdade; Instituição B — Colégio) com dados completamente isolados, mas na mesma infraestrutura. As duas instituições pertencem ao mesmo grupo e compartilham eventuais usuários cross-tenant (ex.: professor do Colégio que também é aluno na Faculdade).

## Decisão

Adotar **multi-tenancy por coluna (`tenant_id`)** em um banco PostgreSQL único com schemas separados por domínio (`ead`, `gestao`, `analytics`).

Toda query que acessa dados de tenant **obrigatoriamente inclui `WHERE tenant_id = ?`**. O `tenant_id` ativo é embutido no JWT (`activeTenantId`) e revalidado server-side em cada Server Action e API Route — nunca confiando no client.

Identidade global separada do vínculo: tabela `users` guarda identidade (email, senha, CPF opcional); tabela `tenant_memberships` guarda o vínculo `userId ↔ tenantId ↔ role`. Um usuário pode ter múltiplos vínculos.

## Alternativas rejeitadas

| Alternativa | Motivo da rejeição |
|---|---|
| Banco separado por tenant | Overhead operacional alto; migrações precisariam rodar N vezes; Railway cobraria por banco adicional. Não justificado com apenas 2 tenants. |
| Schema separado por tenant | Migrações Prisma não suportam múltiplos schemas de forma nativa; complexidade de manutenção alta sem benefício proporcional. |
| Subdomínio + instância separada | Custo de infra inviável; duplicação de código e operação. |

## Consequências

**Positivas:**
- Uma única migration roda para todos os tenants
- Super admin pode alternar entre tenants sem reautenticar
- Custo de infra mínimo — 1 banco, 1 aplicação

**Negativas / riscos:**
- Bug de isolamento é catastrófico (leak cross-tenant) — mitigado por lint rule + testes que verificam presença de `tenant_id` em toda query crítica
- Escala horizontal de tenant específico é mais difícil — aceitável para o volume esperado

## Regras de implementação

1. Todo model Prisma que contém dados de tenant **deve ter campo `tenantId String`**
2. Todo Server Action e API Route **deve extrair `tenantId` do JWT e aplicar no filtro** — nunca aceitar `tenantId` do corpo da requisição
3. Testes de red-team incluem: tentar acessar recurso de tenant B com JWT de tenant A → deve retornar 403
