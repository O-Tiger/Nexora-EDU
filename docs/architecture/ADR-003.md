# ADR-003 — Páginas customizadas com slug livre no Page Builder

**Status:** Proposto — deferido para fase futura  
**Data:** 2026-06-22  
**Autor:** Paulo (O-Tiger)

## Contexto

O Page Builder atual suporta 4 tipos de página fixos (`home`, `course`, `login`, `certificate`) definidos no enum `PAGE_TYPES` em `@nexora/validators`. Há demanda para que admins criem páginas com slugs livres (ex: `/p/sobre-nos`, `/p/parceiros`), publicando conteúdo institucional arbitrário sem precisar de código.

## Decisão

A implementação de páginas customizadas foi **deferida**. Os tipos fixos atendem os casos de uso imediatos. A feature será construída em fase futura, seguindo a estratégia descrita abaixo.

## Estratégia planejada

### Modelo de dados

Adicionar campo `slug` ao modelo `PageLayout`:

```prisma
model PageLayout {
  // ...campos atuais...
  /// null para tipos fixos; slug único por tenant para páginas customizadas
  slug  String?
  @@unique([tenantId, slug])
}
```

Os tipos fixos continuam usando `pageType` (`home`, `course`, etc.) com `slug = null`. Páginas customizadas usam `pageType = "custom"` com `slug` preenchido.

### Validação de slug

- Regex: `/^[a-z0-9]+(?:-[a-z0-9]+)*$/` (lowercase, hífens, sem espaços, sem acentos)
- Tamanho: 3–80 caracteres
- Blocklist: `home`, `course`, `login`, `certificate`, `admin`, `api`, `p`, `login`, `consentimento`
- Unicidade validada server-side na criação — retornar erro genérico se duplicado (não vazar slugs de outros tenants)

### Roteamento público

Nova rota `/p/[slug]/page.tsx` separada de `/p/[pageType]/page.tsx`. Busca por `pageType = "custom"` + `slug` + `tenantId`. Retorna 404 se não publicada.

### UI

- Nova entrada "Páginas customizadas" em `/admin/paginas`
- Botão "Nova página" → modal com campo de título (slug gerado automaticamente, editável)
- Deleção disponível apenas para páginas customizadas (não para tipos fixos)
- Mesmo editor (`PageBuilderEditor`) reaproveitado — `pageType` passado como `"custom"` + novo prop `slug`

### Ações server necessárias

- `createCustomPageAction(title, slug)` — valida slug, cria rascunho vazio
- `deleteCustomPageAction(slug)` — deleta todas as versões do slug
- `listCustomPagesAction()` — lista slugs do tenant com status publicado/rascunho

## Alternativas rejeitadas

**CMS externo (Contentful, Sanity):** overhead de integração, custo, dependência externa. O Page Builder já resolve o caso de uso sem terceiros.

**Rotas via Next.js `app/p/[slug]` dinâmico unificado com tipos fixos:** conflito de namespace — slugs customizados poderiam colidir com `home`, `login`, etc. A separação de rotas é mais segura.

**Slugs com `pageType` livre (sem campo dedicado):** quebraria o `PageTypeSchema` atual do Zod e exigiria validação customizada em todos os pontos que hoje usam o enum.

## Consequências

- Migração Prisma necessária: `slug String? @unique([tenantId, slug])`
- `PageTypeSchema` em `@nexora/validators` permanece enum para tipos fixos; custom pages validam slug separadamente
- Rota `/p/[slug]` deve resolver ambiguidade com ISR/cache — usar `revalidatePath` no publish
- Testes de IDOR obrigatórios: garantir que tenant A não acesse/edite slug de tenant B
