# ADR-002 — Estratégia de Autenticação

**Data:** 2026-06-02  
**Status:** Aceito  
**Autores:** Paulo (O-Tiger)

---

## Contexto

A plataforma precisa de autenticação segura para múltiplos papéis (ALUNO, PROFESSOR, COORDENADOR, ADMIN, SUPER_ADMIN) em múltiplos tenants. Usuários cross-tenant precisam poder alternar de contexto sem reautenticar. A plataforma é uma aplicação Next.js 15.

## Decisão

Usar **NextAuth.js v5** com:

- **Provider:** Credentials (email + senha)
- **Hash de senha:** Argon2 (pacote `argon2`) — mais seguro que bcrypt contra ataques de GPU
- **Sessão:** JWT strategy (stateless, sem tabela de sessão no banco)
- **Access token:** expiração de 8 horas
- **Refresh:** gerenciado via cookie `httpOnly` pelo NextAuth v5 (session rotation automática)
- **JWT payload:** inclui `id`, `activeTenantId`, `role`, `availableTenants`
- **Rate limit de login:** 5 tentativas por IP → bloqueio de 15 minutos via Upstash Redis (Fase 0 — stub; implementado na Fase 1)
- **Mensagens de erro:** genéricas — nunca revelar se o email existe ou não

### Fluxo de seleção de tenant

1. Usuário faz login com email + senha
2. Se tiver apenas 1 tenant ativo → JWT emitido com esse tenant, redirect direto
3. Se tiver múltiplos tenants → frontend exibe seletor inline; usuário escolhe; novo signIn com `tenantId` → JWT emitido com tenant selecionado

### Proteção de rotas

Middleware Next.js (`src/middleware.ts`) verifica JWT e aplica regras de role por prefixo de rota:
- `/admin` → ADMIN, SUPER_ADMIN
- `/coord` → COORDENADOR, ADMIN, SUPER_ADMIN
- `/prof` → PROFESSOR e acima
- `/aluno` → qualquer role autenticado
- Sem JWT → redirect para `/login`
- Role insuficiente → redirect para `/unauthorized`

## Alternativas rejeitadas

| Alternativa | Motivo da rejeição |
|---|---|
| Auth do zero (JWT manual) | Proibido pelas regras do projeto — complexidade de implementar corretamente rotation, revogação e proteção contra ataques |
| Supabase Auth / Auth0 | Dependency externa de terceiro para feature core; NextAuth v5 suficiente para o caso de uso e mantém controle total |
| Database sessions (sessão no banco) | Overhead de leitura no banco em cada request; JWT stateless é suficiente para o volume esperado |
| bcrypt em vez de Argon2 | Argon2 é o padrão recomendado por OWASP (2023+) para novos sistemas; bcrypt é legado |

## Consequências

**Positivas:**
- Sem tabela de sessão no banco — auth não adiciona query em cada request
- Argon2 é state-of-the-art para hashing de senha
- Seletor de tenant totalmente client-side — zero query extra no servidor para exibir as opções

**Negativas / riscos:**
- JWT não pode ser revogado individualmente sem uma deny-list — mitigado por expiração curta de 8h
- Cross-tenant switch exige novo signIn (não apenas atualização do token) — comportamento intencional para segurança

## Regras de implementação

1. **Nunca** armazenar `passwordHash` em logs ou responses
2. Rate limit em `/api/auth/signin` — implementar na Fase 1 com `@upstash/ratelimit`
3. Resposta de erro de login sempre: `"Email ou senha incorretos. Tente novamente."` — independente do motivo real
4. `tenant_id` no JWT **nunca** aceito de input do client — sempre derivado do banco na autenticação
